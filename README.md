# Taskora

Taskora, интерфейсное название «Таскора», — русскоязычный маркетплейс задач и услуг. Сервис проектируется как модульный монолит: заказчик может купить готовую услугу или разместить индивидуальное задание, исполнитель — опубликовать услугу, откликнуться на задание и выполнить заказ внутри платформы.

Проект не копирует Kwork и Studwork по дизайну, текстам, структуре страниц или фирменному стилю.

- введены уровни исполнителей по объективным метрикам (Новичок, Специалист, Профи, Эксперт): уровень зависит от завершенных оплаченных заказов, среднего рейтинга и доли проигранных споров, пересчитывается автоматически и показывается на витрине исполнителей и в публичном профиле;
- проведен рестайлинг интерфейса: self-hosted шрифт Onest, фирменный синий акцент, мягкие радиусы и тени через дизайн-токены Tailwind v4, полировка карточек услуг и блока доверия на странице услуги;
- создание задания оформлено как мастер из трех шагов («О задаче» → «Условия» → «Проверка») с индикатором прогресса и подсказками; новый заказчик без заданий и заказов видит на дашборде онбординг-блок с приглашением разместить первое задание;
- создание услуги оформлено как мастер из трех шагов («Об услуге» → «Цена и пакеты» → «Проверка»); редактор пакетов вынесен в общий компонент и используется мастером и формой редактирования; новый исполнитель без услуг и откликов видит на дашборде онбординг-блок с двумя путями к первому заказу;
- каталог услуг, страницы категорий и витрина исполнителей отдаются постранично по 24 карточки с сохранением фильтров; навигация вынесена в общий компонент Pagination;
- у услуг появились обложки: исполнитель загружает изображение (JPG/PNG/WebP до 5 МБ) в мастере создания или форме редактирования, обложка показывается в карточках каталога с ленивой загрузкой и на странице услуги, без обложки рисуется аккуратный градиентный fallback с категорией; замена обложки опубликованной услуги возвращает ее на модерацию, файлы лежат на public disk;
- дашборды стали «живыми»: заказчик видит плитки с числом заказов в работе, ждущих приемки и оплаты, счетчик откликов и янтарные блоки «Работа сдана — проверьте» (с датой автоприемки) и «Новые отклики»; исполнитель — заказы в работе/на проверке, активные отклики, доступную к выводу сумму и блок «Заказчик запросил доработку»;
- на главной и бирже появились честные счетчики активности (задания за неделю, опубликованные услуги, исполнители с услугами, завершенные заказы) с кэшем на 10 минут; нулевые метрики скрываются, искусственные цифры не используются.

В проекте не подключены:

- реальный платежный шлюз;
- AI-функции;
- email-, push- и WebSocket-уведомления;
- внешние платные API;
- Docker-зависимый запуск.

## Запуск

Установить PHP-зависимости:

```bash
composer install
```

Установить JS-зависимости:

```bash
npm install
```

Создать локальную SQLite-БД, если файла еще нет:

```bash
php -r "file_exists('database/database.sqlite') || touch('database/database.sqlite');"
```

Выполнить миграции и сидеры:

```bash
php artisan migrate:fresh --seed
```

Собрать frontend:

```bash
npm run build
```

Запустить локальный сервер:

```bash
php artisan serve --host=127.0.0.1 --port=8000
```

Открыть:

```text
http://127.0.0.1:8000
```

Для разработки frontend можно отдельно запустить:

```bash
npm run dev
```

## Запуск полного набора тестов локально

Если локальный PHP использует `memory_limit=128M`, полный набор тестов может падать по памяти до завершения PHPUnit. Для полного локального прогона используйте Composer-скрипт, который запускает PHPUnit напрямую с повышенным лимитом:

```bash
composer test:full
```

Эквивалентные прямые команды:

```bash
php -d memory_limit=512M vendor/bin/phpunit
php -d memory_limit=512M artisan test
```

Обычный `composer test` остается доступен для стандартного запуска без изменения лимита памяти. Если `php -d memory_limit=512M artisan test` в локальной среде все равно запускает дочерний PHP с `128M`, используйте `composer test:full` или прямой запуск `vendor/bin/phpunit`.

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/admin/categories` | список категорий, поиск, фильтр активности, счетчики услуг/заданий/видов |
| `GET` | `/admin/categories/create` | форма создания категории |
| `POST` | `/admin/categories` | сохранение категории с автогенерацией уникального slug |
| `GET` | `/admin/categories/{category}/edit` | форма редактирования категории |
| `PATCH` | `/admin/categories/{category}` | обновление категории без автоматической смены slug |
| `POST` | `/admin/categories/{category}/toggle-active` | включение или скрытие категории |
| `POST` | `/admin/categories/{category}/move-up` | поднять категорию в сортировке |
| `POST` | `/admin/categories/{category}/move-down` | опустить категорию в сортировке |
| `GET` | `/admin/task-types` | список видов заданий, поиск, фильтр активности и категории |
| `GET` | `/admin/task-types/create` | форма создания вида задания |
| `POST` | `/admin/task-types` | сохранение вида задания с уникальным slug |
| `GET` | `/admin/task-types/{taskType}/edit` | форма редактирования вида задания |
| `PATCH` | `/admin/task-types/{taskType}` | обновление вида задания |
| `POST` | `/admin/task-types/{taskType}/toggle-active` | включение или скрытие вида задания |
| `POST` | `/admin/task-types/{taskType}/move-up` | поднять вид задания в сортировке |
| `POST` | `/admin/task-types/{taskType}/move-down` | опустить вид задания в сортировке |

Физическое удаление справочников в MVP отключено. Скрытые категории и виды заданий не предлагаются в новых формах и публичных фильтрах, но старые опубликованные задания и услуги продолжают открываться и показывают название связанного справочника. Публичные описания категорий и видов заданий проходят ContactGuard и не сохраняются, если содержат контакты или предложение перейти вне Таскоры.

## Админское Управление Пользователями

Администратор управляет пользователями из защищенного раздела `/admin/users`. Доступ есть только у роли `admin`; `moderator`, `customer`, `performer` и гости не видят раздел и не могут выполнять POST/PATCH-действия напрямую.

Маршруты:

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/admin/users` | список пользователей, поиск, фильтры по роли, статусу, профилю исполнителя и наличию заказов |
| `GET` | `/admin/users/{user}` | карточка пользователя, связанные данные и аудит действий |
| `GET` | `/admin/users/{user}/edit` | форма редактирования имени, email, роли и админской заметки |
| `PATCH` | `/admin/users/{user}` | обновление имени, email, роли и заметки |
| `POST` | `/admin/users/{user}/block` | блокировка пользователя с обязательной причиной |
| `POST` | `/admin/users/{user}/unblock` | разблокировка пользователя |
| `PATCH` | `/admin/users/{user}/admin-note` | отдельное обновление админской заметки |

В MVP нельзя физически удалять пользователей, менять пароль пользователя из админки, входить от имени пользователя или экспортировать базу пользователей. Карточка пользователя не выводит `password`, `remember_token`, reset-токены, beta-пароль, приватные ключи и полные IP-адреса. Последний IP показывается в маскированном виде.

## Админский Просмотр Заказов

Администратор может проверять заказы из read-only раздела `/admin/orders`. Раздел нужен для поддержки beta/staging: найти заказ, увидеть участников, источник, статусы, workspace-сводку, споры, события, payment operations и ledger. В этом разделе нельзя менять статус заказа, удалять заказ, делать возврат, разблокировать средства или запускать реальные платежи.

Маршруты:

| Метод | Маршрут | Назначение |
|---|---|---|
| `GET` | `/admin/orders` | список заказов с поиском, фильтрами, сортировкой и пагинацией |
| `GET` | `/admin/orders/{order}` | read-only карточка заказа, участники, источник, workspace, споры и финансовая сводка |
| `GET` | `/admin/orders/{order}/events` | полный журнал событий заказа с фильтром типа и сортировкой |
| `GET` | `/admin/orders/{order}/ledger` | payment operations, ledger entries и суммы по счетам |

Список поддерживает query-фильтры `q`, `status`, `payment_status`, `source_type`, `has_dispute`, `date_from`, `date_to`, `price_min`, `price_max`, `customer_id`, `performer_id` и `sort`. Карточка заказа показывает только метаданные файлов заказа: оригинальное имя, MIME-тип, размер, статус и автора. Прямые storage paths, download URL, `stored_name`, `disk`, пароли, remember tokens, beta-пароль и полные IP в Inertia props не передаются.

Блокировка переводит `users.status` в `blocked`, заполняет `blocked_at`, `blocked_by` и `block_reason`. Заблокированный аккаунт не может войти; уже авторизованный заблокированный пользователь будет разлогинен на следующем web-запросе. Нельзя заблокировать самого себя и нельзя заблокировать или разжаловать последнего активного администратора. Смена роли, блокировка, разблокировка и обновление админской заметки пишутся в `user_admin_events`.

## Удержание Оплаты И Проверка

В MVP реальный платежный шлюз не подключен. Действие "Оплатить (заглушка)" переводит заказ в работу, ставит `payment_status = held` и создает внутреннюю операцию `payment_hold` с ledger-записями.

Платежная архитектура хранит только внутреннюю историю:

- `payment_operations` — операции удержания, разблокировки, возврата, комиссии и будущих webhook-событий;
- `ledger_entries` — неизменяемые записи движения средств по счетам `customer_payment`, `escrow`, `performer_pending`, `performer_available`, `platform_fee`, `customer_refund`;
- `provider_webhook_events` — заготовка под будущие события провайдера без endpoint;
- `payout_requests` — заготовка заявок на вывод без банковских реквизитов и реальных выплат.

Реальные карты, платежные реквизиты, паспортные данные, ИНН, СНИЛС и KYC-документы не собираются.

Когда исполнитель отправляет работу, заказ переходит в `submitted_for_review`, запускается срок проверки `review_hold_days` с дефолтом 10 дней, заполняются `review_hold_started_at`, `review_hold_until` и `auto_release_at`. Оплата остается удержанной.

Заказчик может принять работу досрочно кнопкой "Принять работу и разблокировать оплату". В этом случае заказ становится `completed`, оплата переходит в `released`, заполняются `released_at` и `release_reason = customer_early_accept`.

При досрочной приемке, auto-release и решении спора в пользу исполнителя создается операция `release_to_performer`, а ledger переносит сумму исполнителя из `performer_pending` в `performer_available`. При решении спора с возвратом создается `refund_to_customer`, ledger отражает возврат заказчику и сторнирует комиссию через `platform_fee_reverse`.

Если заказчик запрашивает доработку, заказ переходит в `revision_requested`, оплата остается удержанной, а срок проверки очищается. Повторная сдача работы запускает новый срок проверки.

Для автоматической разблокировки просроченных проверок используется команда:

```bash
php artisan orders:release-due
```

В production ее нужно поставить в scheduler/cron, например запускать каждые несколько минут. Команда завершает только заказы `submitted_for_review` с `payment_status = held` и истекшим `review_hold_until`; спорные, отмененные и уже завершенные заказы не изменяются.

## Споры И Арбитраж

Спор можно открыть только по оплаченному заказу в статусах `in_progress`, `submitted_for_review` или `revision_requested`. Спор переводит заказ в `disputed`, оставляет `payment_status = held` и очищает `review_hold_until`/`auto_release_at`, чтобы автоматическая разблокировка оплаты не сработала до решения модератора.

Открыть спор могут только участники заказа: заказчик-владелец или исполнитель. Гости, чужие пользователи, модераторы и администраторы не открывают спор от имени участника. После `completed` и `payment_status = released` спор в MVP не открывается; пост-гарантийный спор оставлен как будущий TODO.

Модератор или администратор видит очередь `/moderator/disputes`, может взять спор в работу и принять одно из решений:

- `release_to_performer` — заказ завершается, оплата разблокируется исполнителю, `release_reason = dispute_release_to_performer`;
- `refund_to_customer` — заказ отменяется, статус оплаты становится `refunded`;
- `return_to_revision` — заказ возвращается в `revision_requested`, оплата остается удержанной.

Сообщения в споре проходят через ContactGuard. При обнаружении email, телефона, Telegram или другого контакта сообщение не сохраняется, создается `moderation_flag` и событие `contact_blocked`.

## Внутренние Уведомления

Уведомления хранятся в стандартной Laravel-таблице `notifications` через database channel. Push, WebSocket, Broadcasting, Reverb и Pusher не подключены.

Ключевые события дополнительно дублируются на email пользователя через mail channel: новый отклик (`task_offer.created`), оплата заказа (`order.payment_held`), сдача работы (`order.work_submitted`), открытие спора (`dispute.opened`) и решение спора (`dispute.resolved`). Список задан в `PlatformNotification::EMAIL_EVENT_TYPES`. Email-дублирование выключается переменной `TASKORA_EMAIL_NOTIFICATIONS_ENABLED=false`. Локально и на staging используется `MAIL_MAILER=log`, поэтому письма попадают в `storage/logs/mail.log`, а не во внешнюю почту; текст письма и служебные строки шаблона локализованы через `lang/ru.json`. Реальный SMTP-провайдер перед public launch подключается отдельной задачей.

Payload уведомления хранится в `data` JSON и содержит:

- `type`;
- `title`;
- `body`;
- `url`;
- `icon`;
- `severity`: `info`, `success`, `warning`, `danger`;
- `related_type`;
- `related_id`;
- `meta`.

В интерфейсе авторизованный пользователь видит счетчик непрочитанных уведомлений в общем dashboard layout. Страница `/notifications` показывает список уведомлений, состояние прочитано/непрочитано, дату, тип, текст, ссылку "Открыть", кнопку "Отметить прочитанным" и действие "Отметить все прочитанными".

Уведомления создаются для событий:

- одобрение и отклонение услуги модератором;
- новый отклик на задание, выбор отклика и отклонение отклика;
- создание заказа из услуги или отклика;
- локальная оплата заказа;
- сдача работы, запрос доработки, завершение заказа и auto-release;
- новое сообщение или файл в рабочей области заказа;
- открытие спора, взятие спора в работу, новое сообщение в споре и решение спора.

Сервис `NotificationService` не отправляет уведомление пользователю-инициатору для пользовательских сообщений и дедуплицирует получателей в рамках одного события. Системные события завершения заказа и auto-release уведомляют обе стороны заказа.

## Messages v2

Страница `/messages` доступна авторизованным пользователям и объединяет переписки по заказам и спорам без WebSocket, Broadcasting, Reverb, Pusher и внешних сервисов. Messages v2 оформлен как рабочий messenger: слева список диалогов, в центре активный чат, справа контекст заказа или спора. На mobile список и чат открываются как отдельные состояния, а детали сделки доступны в раскрываемом блоке.

Что реализовано:

- вкладки «Все», «Непрочитанные», «Заказы», «Споры»;
- поиск по названию заказа и имени участника; email не используется в поиске для обычных заказчика и исполнителя;
- фильтр по статусу заказа;
- сортировка «Сначала новые» и «Сначала непрочитанные»;
- карточки диалогов с участником, статусом, последним сообщением, unread badge и выбранным состоянием;
- отдельные страницы `/messages/orders/{order}` и `/messages/disputes/{dispute}`;
- sticky header активного диалога, safety banner, date separators, bubbles текущего пользователя справа и сообщения собеседника слева;
- компактная карточка сделки, последние ключевые события, последние файлы и правая панель с участниками, оплатой, сроками и быстрыми ссылками;
- файлы отображаются только как безопасные метаданные и скачиваются через существующий контроллер с проверкой доступа;
- отправка сообщений через те же `StoreOrderMessageRequest`, `StoreDisputeMessageRequest`, policies, `ContactGuard`, `OrderEventLogger` и database notifications, что и в workspace/спорах;
- отметка прочтения в таблице `conversation_reads`;
- общий счетчик непрочитанных сообщений в `DashboardLayout` и `PublicLayout`.

Непрочитанными считаются сообщения после `last_read_at`, отправленные другими участниками. Открытие диалога автоматически обновляет `last_read_at`. В shared Inertia props передается только `messages.unread_count`, без списка сообщений.

Доступ:

- заказчик и исполнитель видят только свои заказные диалоги;
- модератор и администратор видят переписку по спорам через `/messages/disputes/{dispute}`;
- модератор и администратор не получают доступ к прямому чату заказа без спора;
- Inertia payload не должен содержать `password`, `remember_token`, reset-токены, beta-пароль или приватные пути файлов заказа.

## Rate Limits И Ошибка 429

Для MVP включены прикладные ограничения на частые действия:

| Действие | Ограничение |
|---|---|
| сообщения заказа и спора | до 20 в минуту на пользователя |
| загрузка файлов заказа | до 10 за 10 минут на пользователя |
| создание задания, услуги или спора | до 10 в час на пользователя |
| отклики на задания | до 30 в час на пользователя |
| маршруты уведомлений | до 60 в минуту на пользователя |

При превышении лимита пользователь получает русскоязычную страницу `429` с объяснением, что действие временно ограничено.

| `pending_review` | На модерации |
| `published` | Опубликована |
| `rejected` | Отклонена |
| `archived` | В архиве |

## Заказы И Заглушка Оплаты

Заказ создается двумя способами:

- из опубликованной услуги через `POST /services/{service:slug}/order`;
- из выбранного отклика через `POST /customer/task-offers/{offer}/accept`.

При создании заказа рассчитываются:

- `price` — цена выбранного пакета услуги или отклика;
- `platform_fee_amount` — локальная комиссия платформы по `TASKORA_PLATFORM_FEE_PERCENT`;
- `performer_amount` — сумма исполнителю после комиссии.

Реальный платежный шлюз не подключен. Кнопка «Оплатить (заглушка)» переводит заказ из `awaiting_payment` в `in_progress`, а `payment_status` из `unpaid` в `held`. Это нужно только для локальной проверки жизненного цикла.

Статусы заказа:

| Технический статус | Русский статус |
|---|---|
| `awaiting_payment` | Ожидает оплаты |
| `in_progress` | В работе |
| `submitted_for_review` | На проверке |
| `revision_requested` | Требуется доработка |
| `completed` | Завершен |
| `disputed` | Спор |
| `canceled` | Отменен |

Статусы оплаты:

| Технический статус | Русский статус |
|---|---|
| `unpaid` | Не оплачен |
| `held` | Оплата удерживается |
| `released` | Оплата разблокирована |
| `refunded` | Возврат |
| `canceled` | Отменена |

## Отзывы И Доверие

Отзыв можно оставить только по своему заказу после `status = completed` и `payment_status = released`. Один заказ дает только один отзыв. Исполнитель не может оставить отзыв самому себе, а заказчик не может передать в запросе чужие `service_id`, `task_id` или `performer_id`: эти связи берутся из заказа.

Правила отзыва:

- `rating` обязателен и принимает значения от 1 до 5;
- `comment` необязателен, максимум 2000 символов;
- `ContactGuard` проверяет комментарий до сохранения;
- при найденном контакте отзыв не сохраняется, создается `moderation_flag` с причиной `contact_detected_in_review`;
- в MVP редактирование отзывов и админское скрытие отзывов оставлены как TODO.

Публичные рейтинги не являются заглушками. `services.rating`, `services.reviews_count`, `services.orders_count`, `users.performer_rating`, `users.performer_reviews_count` и `users.performer_completed_orders_count` пересчитываются через `ReviewAggregateService` по опубликованным публичным отзывам и завершенным оплаченным заказам. Если отзывов нет, интерфейс показывает «Нет отзывов», а не искусственный рейтинг.

После завершения заказа заказчик получает уведомление «Заказ завершен — оставьте отзыв исполнителю». После публикации отзыва исполнитель получает уведомление «Вам оставили новый отзыв». Уведомления остаются внутренними database notifications.

## Уровни Исполнителей

Уровень исполнителя считается только по объективным метрикам и пересчитывается автоматически при завершении заказа, публикации отзыва и решении спора. Купить или вручную выставить уровень нельзя.

| Уровень | Завершенные оплаченные заказы | Рейтинг | Доля проигранных споров |
|---|---|---|---|
| Новичок | до 3 | любой | — |
| Специалист | от 3 | нет отзывов или ≥ 4.0 | ≤ 20% |
| Профи | от 10 | ≥ 4.5 | ≤ 10% |
| Эксперт | от 30 | ≥ 4.7 | ≤ 5% |

Проигранным считается спор с решением «средства возвращены заказчику». Уровень показывается на витрине `/performers` и в публичном профиле исполнителя; правила заданы в `App\Support\PerformerLevel`.

## Рабочая Область Заказа

Рабочая область доступна только двум участникам заказа:

- заказчику-владельцу заказа;
- исполнителю заказа.

Модератор и администратор пока не получают доступ к рабочей области заказа. Для будущего доступа нужен отдельный модуль споров и арбитража.

В рабочей области есть:

- верхний блок заказа со статусом, суммой, сроком и участниками;
- предупреждение о локальной заглушке оплаты;
- предупреждение для неоплаченного заказа;
- чат заказа;
- загрузка и список файлов;
- история событий заказа;
- быстрые действия по текущему статусу.

Сообщения проходят через `ContactGuard`. Если в сообщении найден email, телефон, Telegram, WhatsApp, VK, Discord, Skype, банковская карта или фраза про уход с платформы, сообщение не сохраняется, создаются `moderation_flag` и событие `contact_blocked`.

Файлы хранятся на приватном диске `local` в `storage/app/private`, без прямой публичной ссылки. Скачивание идет только через controller route с проверкой участника заказа.

Разрешенные типы файлов:

- `pdf`, `doc`, `docx`;
- `xls`, `xlsx`, `csv`;
- `ppt`, `pptx`;
- `txt`;
- `png`, `jpg`, `jpeg`, `webp`;
- `zip`.

Ограничение размера: до 20 MB. `rar` не добавлен в MVP, чтобы не зависеть от нестабильной MIME-валидации в локальной PHP-среде.

Файлы заказа хранятся на приватном диске `local` в каталоге `orders/{order_id}`. Они не должны попадать в public storage и не должны открываться через `/storage/...`. Скачивание идет только через маршруты `.../files/{file}/download`, где контроллер проверяет участника заказа через policy.

Проверка ContactGuard выполняется по названию файла. Для `txt` и `csv` дополнительно проверяется текстовое содержимое. Полноценный парсинг `docx`, `pdf`, `xlsx`, `pptx` и OCR изображений запланированы на будущий этап без добавления внешних зависимостей сейчас.

События заказа пишутся в `order_events` для:

- создания заказа;
- оплаты через локальную заглушку;
- сдачи работы;
- запроса доработки;
- завершения;
- отмены;
- отправки сообщения;
- загрузки файла;
- блокировки контактов.

## Защита От Передачи Контактов

При создании и редактировании услуги проверяются название, короткое описание, полное описание, названия и описания пакетов. При создании и редактировании задания проверяются название и описание. При отправке отклика проверяется сообщение исполнителя. В рабочей области проверяются сообщения, названия файлов и содержимое `txt`/`csv`. Блокируются явные email, телефоны, Telegram, WhatsApp, VK, Discord, Skype, банковские карты и фразы вроде "напиши в тг", "давай напрямую", "оплата мимо сайта".

Если правило сработало, действие не сохраняется, пользователь получает ошибку на русском языке, а срабатывание записывается в `moderation_flags` с `entity_type` услуги, задания или отклика.

|---|---|---|
| `GET` | `/beta-testing` | страница с правилами, тестовыми аккаунтами и чек-листами по ролям |
| `GET` | `/beta-feedback/create` | форма отправки beta-обращения для гостя или авторизованного пользователя |
| `POST` | `/beta-feedback` | сохранение beta-обращения во внутреннюю таблицу |
| `GET` | `/admin/beta-feedback` | административная очередь beta-отзывов |
| `GET` | `/admin/beta-feedback/{feedback}` | карточка beta-отзыва |
| `POST` | `/admin/beta-feedback/{feedback}/status` | смена статуса beta-отзыва |

## Восстановление пароля

На странице входа есть ссылка `/forgot-password`. Гость вводит email, а приложение всегда показывает нейтральное сообщение: если аккаунт существует, ссылка для сброса будет отправлена. Это не раскрывает, зарегистрирована ли почта.

Маршруты password reset:

| Метод | Маршрут | Имя | Назначение |
|---|---|---|---|
| `GET` | `/forgot-password` | `password.request` | форма запроса ссылки |
| `POST` | `/forgot-password` | `password.email` | отправка reset link через Laravel password broker |
| `GET` | `/reset-password/{token}` | `password.reset` | форма нового пароля |
| `POST` | `/reset-password` | `password.store` | сохранение нового пароля |

В `local` и `staging` используется `MAIL_MAILER=log`, поэтому письмо не отправляется во внешнюю почту: reset link пишется в `storage/logs/mail.log`. Реальный email-провайдер перед public launch нужно подключить отдельной задачей.

Администратор может проверить текущий почтовый режим на `/admin/mail-settings`. Страница не отправляет письма, не подключает внешний SMTP и не показывает `MAIL_PASSWORD`, SMTP-токены или другие секретные значения из `.env`.

## Закрытое beta-тестирование

Staging-версия доступна по адресу `https://staging.таскора.рф` и закрыта общим beta-паролем. Beta-пароль не хранится в репозитории, README или документации; его нужно передавать тестировщикам отдельно через личный канал.

Короткая инструкция для друзей лежит в [docs/beta-friends-message.md](docs/beta-friends-message.md). Реальные платежи, выплаты и платежные провайдеры на beta/staging не подключены: все оплаты работают только как заглушка. Обратная связь собирается через `/beta-feedback/create`.

Для production нужно явно выставить:

```dotenv
APP_ENV=production
APP_DEBUG=false
APP_URL=https://example.com
```

Scheduler в production должен запускаться через cron:

```bash
* * * * * cd /path/to/taskora && php artisan schedule:run >> /dev/null 2>&1
```

## Светлая и темная тема

Интерфейс поддерживает два режима оформления: светлая тема и темная тема. Выбор хранится в `localStorage` по ключу `taskora_theme` и применяется на уровне `<html>` через class-based dark mode Tailwind. Старое сохраненное значение `system` считается устаревшим и при загрузке переводится в светлую тему.

До загрузки React базовый Blade layout выполняет небольшой pre-mount script: он читает сохраненную настройку и сразу выставляет или снимает класс `dark`. Это уменьшает мигание неправильной темы без зависимости от `matchMedia`.

Переключатель темы находится в публичном и кабинетном layout, а также на beta gate. Это компактная switch-кнопка без выпадающего списка: клик сразу переключает светлую и темную тему. Реальные платежи, внешние сервисы и production-настройки с темой не связаны.

Визуальный стиль публичных страниц строится на спокойной светлой/темной палитре, фирменном синем акценте и умеренных иконках из `lucide-react`. Основной шрифт — Onest (variable, self-hosted в `public/fonts/onest-variable.woff2`, лицензия OFL рядом в `onest-OFL.txt`), подключен через `@font-face` с `font-display: swap` и preload в Blade layout; внешние CDN для шрифтов не используются. Дизайн-токены заданы в `resources/css/app.css` через Tailwind v4 `@theme`: палитра акцента, радиусы скруглений и тень карточек меняются в одном месте без правки компонентов.

Favicon assets лежат в `public`: `favicon.ico`, `favicon-32x32.png`, `favicon-192x192.png` и `apple-touch-icon.png`. Browser favicon используют прозрачный фон и cache-busting в Blade layout; `apple-touch-icon.png` остается отдельным непрозрачным вариантом для iOS, чтобы система не подставляла нежелательный фон.
