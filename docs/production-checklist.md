# Production Checklist

Этот список фиксирует, что нужно проверить перед публичным запуском Taskora. MVP пока работает без реального платежного шлюза, внешней почты, push, WebSocket и ИИ-функций.

## Environment

- Установить `APP_ENV=production`.
- Установить `APP_DEBUG=false`.
- Установить реальный `APP_URL=https://example.com`.
- Сгенерировать и сохранить production `APP_KEY`.
- Проверить `APP_LOCALE=ru`, `APP_FALLBACK_LOCALE=ru`.
- Не переносить локальные demo-аккаунты и demo-контент в production.

## Database

- Для MVP локально используется SQLite.
- Перед публичным запуском подготовить PostgreSQL и миграцию данных только отдельной задачей.
- Проверить резервное копирование БД.
- Проверить индексы для заказов, уведомлений, споров, сообщений и файлов.
- Не запускать `migrate:fresh --seed` на production.

## Storage

- Файлы заказа хранить приватно на `local` disk или будущем защищенном S3-диске.
- Не хранить `order_files` в `public` disk.
- `php artisan storage:link` должен открывать только `storage/app/public` для будущих публичных изображений.
- Проверить, что `/storage/orders/...` не открывает файлы заказа.
- Скачивание файлов заказа должно идти только через контроллер с `auth` и policy.

## Scheduler

- Проверить, что `orders:release-due` есть в `php artisan schedule:list`.
- Настроить cron:

```bash
* * * * * cd /path/to/taskora && php artisan schedule:run >> /dev/null 2>&1
```

- Убедиться, что `orders:release-due` запускается hourly.
- Проверить, что спорные заказы `disputed` не разблокируются автоматически.

## Queue

- В MVP используется `QUEUE_CONNECTION=sync`.
- Перед высокой нагрузкой перейти на `database` или `redis` отдельной задачей.
- После перехода добавить worker, supervisor/systemd и мониторинг failed jobs.
- Не подключать внешние очереди без отдельного решения по инфраструктуре.

## Security

- Проверить `APP_DEBUG=false`.
- Проверить rate limits для сообщений, файлов, создания сущностей, откликов и уведомлений.
- Проверить, что нет `_codex-login`, `smoke-login` и временных debug routes.
- Проверить доступ ролей `customer`, `performer`, `moderator`, `admin`.
- Проверить, что чужие заказы, споры, уведомления и файлы недоступны.
- Проверить ContactGuard на сообщениях, откликах, заданиях, услугах и файлах.
- Проверить русские страницы `403`, `404`, `429`, `500`, `503`.

## Monitoring

- Настроить сбор Laravel logs.
- Добавить мониторинг HTTP 500/503.
- Добавить мониторинг scheduler.
- Добавить мониторинг заполнения диска storage.
- Добавить алерт на рост 429, 403 и ошибок загрузки файлов.

## Payments

- В MVP реальный платежный шлюз не подключен.
- `TASKORA_PAYMENTS_MODE=stub` должен оставаться активным до отдельного решения владельца.
- Перед подключением платежей нужна отдельная задача: провайдер, договор, вебхуки, фискализация, возвраты и спорные сценарии.
- Не хранить платежные реквизиты пользователей в текущей MVP-схеме.

## Legal

- Подготовить пользовательское соглашение.
- Подготовить политику конфиденциальности.
- Описать правила модерации и запрет передачи контактов.
- Описать правила споров, возвратов и удержания средств до подключения реального платежного шлюза.
- Проверить требования к персональным данным и хранению файлов.

## Before Public Launch

- Пройти `php artisan test`.
- Пройти `npm run build`.
- Пройти `composer validate --strict`.
- Проверить `php artisan route:list --except-vendor`.
- Проверить `php artisan schedule:list`.
- Проверить основные публичные страницы и кабинеты на 360, 390, 768 и 1280 px.
- Проверить регистрацию, вход, создание задания, отклик, заказ, чат, файл, спор и уведомления.
- Удалить или отключить demo-данные.
- Подготовить резервное копирование и план отката.
