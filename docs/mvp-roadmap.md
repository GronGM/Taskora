# MVP Roadmap

Разработка Taskora должна идти по этапам. Цель — не собрать все сразу, а последовательно получить проверяемый модульный монолит.

## Этап 0. Окружение И Каркас

Цель: подготовить минимальный Laravel + Inertia + React проект.

Статус: выполнено для локального запуска на SQLite. PostgreSQL и Redis оставлены на следующий инфраструктурный этап, чтобы не блокировать разработку MVP.

Задачи:

- установить PHP, Composer и Git;
- создать Laravel-приложение;
- подключить Inertia + React;
- подключить Tailwind CSS и Vite;
- настроить `.env` на SQLite;
- добавить базовые layout-компоненты;
- добавить русскую локаль интерфейса;
- зафиксировать baseline в Git.

Результат:

- приложение запускается локально;
- открывается стартовая страница;
- проходят базовые тесты.

## Этап 1. Авторизация И Роли

Цель: пользователи могут регистрироваться, входить и получать роль.

Статус: выполнена минимальная рабочая версия.

Задачи:

- регистрация;
- вход и выход;
- роли `customer`, `performer`, `moderator`, `admin`;
- role middleware для dashboard-маршрутов;
- кабинеты заказчика, исполнителя, модератора и администратора;
- локальные seed-пользователи для проверки.

Результат:

- пользователь входит в систему;
- видит интерфейс своей роли;
- не может открыть чужую защищенную область.

## Этап 2. Публичная Часть

Цель: сформировать основу витрины Taskora.

Статус: выполнена основа публичной витрины: главная, каталог услуг, страницы категорий, страницы услуг и витрина исполнителей. Каталог берет категории и опубликованные demo-услуги из базы.

Задачи:

- главная страница;
- каталог услуг;
- страница категории;
- карточка услуги;
- профиль исполнителя;
- биржа заданий;
- страницы входа и регистрации;
- заглушки правил и политики конфиденциальности.

Результат:

- публичные страницы доступны без входа;
- интерфейс на русском языке;
- структура не копирует Kwork и Studwork.
- опубликованные demo-услуги видны в каталоге, черновики скрыты.

## Этап 3. Категории И Услуги

Цель: исполнитель может создать услугу, а заказчик увидеть ее в каталоге.

Статус: выполнена основа категорий, публичного каталога, управления услугами исполнителя и модераторской очереди услуг. Исполнитель может создать услугу, отредактировать ее, отправить на модерацию и архивировать. Модератор или администратор может одобрить услугу, отклонить ее с причиной и обработать флаги модерации.

Задачи:

- CRUD категорий для админа;
- создание и редактирование услуги в кабинете исполнителя;
- опции услуги;
- статусы публикации;
- модерация услуги;
- фильтры каталога;
- публичная карточка услуги.

Результат:

- услуга создается исполнителем;
- отправляется на модерацию;
- появляется в каталоге.

Оставшаяся работа этапа:

- CRUD категорий для админа.

## Этап 4. Индивидуальные Задания И Отклики

Цель: заказчик может создать задание, а исполнитель откликнуться.

Статус: выполнена рабочая основа индивидуальных заданий и откликов. Заказчик может создавать черновики, публиковать и архивировать задания, видеть отклики и отклонять их. Исполнитель может отправить один отклик на опубликованное задание, увидеть список своих откликов и отозвать отправленный отклик. Публичная биржа `/tasks` показывает только опубликованные задания.

Задачи:

- создание задания — выполнено;
- список "мои задания" — выполнено;
- публичная биржа заданий — выполнено;
- страница задания — выполнено;
- отклики исполнителей — выполнено;
- выбор отклика заказчиком — выполнено;
- модерация заданий и откликов через ContactGuard — выполнена базовая блокировка контактов и запись `moderation_flags`.

Результат:

- задание публикуется;
- исполнитель отправляет отклик;
- заказчик видит отклики и может отклонить неподходящий отклик.

Оставшаяся работа этапа:

- ручная очередь модерации заданий, если понадобится сверх базовой блокировки ContactGuard.

## Этап 5. Заказы И Статусы

Цель: создать единый заказный процесс для услуги и задания.

Статус: выполнена минимальная рабочая версия без реального платежного шлюза. Заказ создается из опубликованной услуги или принятого отклика, проходит локальную заглушку оплаты, сдачу работы, срок проверки, доработку, досрочную приемку, автоматическую разблокировку оплаты по истечении проверки и отмену до оплаты.

Задачи:

- заказ из готовой услуги — выполнено;
- заказ из выбранного отклика — выполнено;
- статусная модель заказа — выполнено;
- action-классы для переходов статусов;
- таймлайн заказа;
- заглушка платежей — выполнено локально без шлюза;
- удержание оплаты на срок проверки — выполнено;
- досрочная разблокировка оплаты заказчиком — выполнено;
- автоматическая разблокировка командой `orders:release-due` — выполнено;
- расчет комиссии платформы — выполнено.

Результат:

- заказ создается из двух сценариев;
- статус меняется только разрешенными переходами;
- оплата работает как локальная заглушка без подключения шлюза;
- после сдачи работы оплата остается удержанной до принятия, доработки, будущего спора или авторазблокировки.

Оставшаяся работа этапа:

- вынести переходы статусов в action-классы;
- реализовать споры и арбитраж, чтобы спор останавливал авторазблокировку оплаты;
- позже заменить локальную заглушку отдельной платежной записью без подключения реального шлюза до отдельного решения владельца.

## Этап 6. Чат И Файлы

Цель: заказчик и исполнитель работают внутри заказа.

Статус: выполнена минимальная рабочая версия. Участники заказа могут открыть workspace, переписываться, загружать приватные файлы, скачивать их через контроллер с проверкой доступа и видеть историю системных событий заказа.

Задачи:

- чат заказа — выполнено;
- загрузка файлов — выполнено;
- ограничения типов и размеров файлов — выполнено;
- policies для доступа к файлам — выполнено через `OrderPolicy`;
- системные сообщения о смене статусов — выполнено через `order_events`;
- базовые уведомления — выполнено через database notifications.

Результат:

- участники заказа могут переписываться;
- файлы доступны только участникам заказа.

Оставшаяся работа этапа:

- ручная модерация файлов;
- будущая проверка содержимого документов и OCR без подключения ИИ в MVP.

## Этап 7. Антиувод Контактов

Цель: закрыть риск передачи контактов и ухода сделки с платформы.

Статус: начата базовая реализация. `ContactGuard` проверяет услуги, задания, отклики, сообщения заказа, названия файлов и содержимое `txt`/`csv`, блокирует явные контакты и сохраняет `moderation_flags`. Открытые флаги доступны модератору и могут быть отмечены обработанными.

Задачи:

- проверка чата перед отправкой — выполнено;
- проверка заданий и откликов — выполнено;
- проверка профилей;
- проверка названий файлов — выполнено для файлов заказа;
- проверка содержимого текстовых файлов — выполнено для `txt`/`csv`;
- экран флагов в админке.

Результат:

- явные нарушения блокируются;
- спорные случаи уходят на ручную модерацию;
- все срабатывания сохраняются.

## Этап 8. Админка, Модерация, Споры

Цель: дать команде инструменты управления платформой.

Статус: споры и арбитраж выполнены в MVP-объеме. Участники заказа могут открыть спор по оплаченному заказу, спор останавливает auto-release, модератор или администратор видит очередь, берет спор в работу, переписывается со сторонами и принимает решение по удержанной оплате.

Задачи:

- админ-дэшборд;
- управление пользователями;
- модерация услуг;
- модерация заданий;
- список заказов;
- список споров — выполнено;
- карточка спора — выполнено;
- решение спора — выполнено;
- просмотр moderation flags.

Результат:

- модератор может обработать спорные материалы;
- админ видит ключевые сущности платформы.

Оставшаяся работа этапа:

- полноценная админская панель управления пользователями и заказами;
- частичный возврат, частичная выплата исполнителю и распределение комиссии по спору;
- расширенная модерация заданий.

## Этап 9. Внутренние Уведомления

Цель: пользователь видит важные события внутри платформы без email, push и WebSocket.

Статус: выполнена MVP-версия. Уведомления хранятся в таблице `notifications`, создаются через `NotificationService` и `PlatformNotification`, передаются в layout через Inertia shared props и доступны на странице `/notifications`.

Задачи:

- database notifications — выполнено;
- универсальный notification-класс — выполнено;
- единый сервис уведомлений — выполнено;
- счетчик непрочитанных уведомлений — выполнено;
- список уведомлений — выполнено;
- отметить одно уведомление прочитанным — выполнено;
- отметить все уведомления прочитанными — выполнено;
- уведомления по услугам, откликам, заказам, сообщениям, файлам и спорам — выполнено;
- feature-тесты ключевых сценариев — выполнено.

Результат:

- пользователь видит события в интерфейсе;
- уведомления не уходят во внешние каналы;
- чужие уведомления нельзя открыть или отметить прочитанными.

## Этап 10. Отзывы И Доверие

Цель: добавить базовые сигналы качества.

Статус: выполнена MVP-версия. Заказчик может оставить один отзыв после завершенного и разблокированного заказа, комментарий проходит ContactGuard, рейтинги услуг и исполнителей пересчитываются из реальных отзывов, а публичные страницы показывают реальные доверительные сигналы без заглушек.

Задачи:

- отзыв после завершения заказа — выполнено;
- рейтинг исполнителя — выполнено через `users.performer_rating`;
- счетчик выполненных заказов — выполнено через `users.performer_completed_orders_count` и `services.orders_count`;
- публичные сигналы доверия — выполнено в карточках услуг, странице услуги и `/performers`;
- публичная история отзывов — выполнено на `/performers/{user}/reviews`;
- защита от передачи контактов в отзыве — выполнено через `ContactGuard` и `moderation_flags`.

Результат:

- заказчик может оставить отзыв;
- витрина исполнителей показывает базовую репутацию;
- услуги не показывают искусственный рейтинг при отсутствии отзывов.

## Этап 11. Тесты И Полировка

Цель: довести MVP до проверяемого состояния.

Статус: UX-polish и production-readiness выполнены для текущего MVP-слоя. Добавлены адаптивные правки layout, русские error pages, rate limits, scheduler для `orders:release-due`, production checklist и тесты ключевых production-контрактов.

Задачи:

- тесты policies;
- тесты статусных переходов;
- тесты антиувода контактов;
- smoke-тесты публичных страниц;
- адаптив — выполнено на уровне общих layout и основных страниц;
- пустые состояния — выполнено в ключевых списках и workspace;
- ошибки форм — базовые русские ошибки сохранены;
- визуальная полировка — выполнена для MVP-слоя;
- production checklist — выполнено;
- rate limits и страница `429` — выполнено;
- scheduler для `orders:release-due` — выполнено.

Результат:

- основные сценарии покрыты тестами;
- интерфейс пригоден для демонстрации MVP.

## Правила Разработки

- Не подключать платежный шлюз в MVP.
- Не добавлять ИИ-функции в MVP.
- Не добавлять зависимости без необходимости.
- Не копировать дизайн и тексты конкурентов.
- Перед крупными изменениями сначала фиксировать план.
- После каждого этапа писать: что сделано, какие файлы изменены, как проверить, какой следующий шаг.

## Следующий Этап

Следующий приоритет MVP — профили и верификация исполнителей: расширить публичную страницу исполнителя, добавить проверяемые поля опыта, подготовить будущую модерацию профиля и не подключать ИИ, платежные сервисы или крупные зависимости.
